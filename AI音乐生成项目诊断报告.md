# AI 音乐生成项目诊断报告

> **核心结论：项目目前不具备生成真实演唱歌曲的能力**
>
> 工程骨架设计合理，Pipeline 结构、模块划分、API 设计值得保留。核心问题在于人声合成技术路径选型错误，需要在关键环节做根本性替换。

---

## 问题总览

| 问题 | 严重程度 | 是否可修复 |
|------|---------|-----------|
| Fish Speech 不能唱歌，只能说话 | 🔴 致命 | 需更换模型 |
| 人声与旋律完全无对齐 | 🔴 致命 | 需重构流程 |

## 🔴 致命问题

### 1. Fish Speech 根本不是唱歌模型

这是整个项目最核心的问题。Fish Speech 是一个 **TTS（文字转语音）模型**，其设计目标是生成自然语音，而不是演唱。

**具体缺陷：**

- 没有音高跟踪能力，无法让人声跟随旋律音高走
- 没有节拍对齐能力，无法让字/词卡在节拍点上
- 不理解乐句、呼吸点、换气位置
- 输出的是朗读腔，不是歌唱腔

**解决方案：** 应替换为 **SVS（Singing Voice Synthesis）模型**，例如 DiffSinger、VITS-SVS，或使用 **SVC（Singing Voice Conversion）** 对已有人声做音色转换。

---

### 2. 人声与伴奏完全脱节

当前生成流程如下：

```
ACE-Step 生成伴奏
        +
Fish Speech 生成人声（说话）
        ↓
     叠加混音
```

这两个过程**互不感知**：

- 人声不知道伴奏的调性（Key）
- 人声不知道伴奏的节拍（BPM）
- 人声不跟随任何旋律走向

混音后的结果是「一段说话声 + 一段伴奏」，而不是真正意义上的演唱歌曲。

---

### 3. 旋律生成是占位符，完全未实现

文档中已明确承认：

> "旋律生成功能不完善（只是占位符）"

`melody_generator.py` 仅生成了一个 MIDI 文件，但这个旋律**完全没有被用来引导人声合成**。`_synthesize_vocal()` 函数虽然接受 `melody_path` 参数，却没有任何代码将旋律信息注入 Fish Speech 的生成过程。

```python
# 当前代码 - melody_path 参数完全被忽略
def _synthesize_vocal(lyrics, melody_path, voice_profile, song_id):
    embedding = torch.load(voice_profile['prompt_embedding_path'])
    audio = fish_speech.synthesize(
        text=lyrics,
        prompt_embedding=embedding,  # 只用了声音嵌入，旋律信息丢失
        temperature=0.8,
        top_p=0.9
    )
```

## 🟠 严重问题

### 4. 风格推荐是硬编码假数据

`recommendation_engine.py` 中的置信度分数是写死的常量，不是真正的声音分析结果：

```python
scores = {
    "pop_ballad": 0.85,
    "folk_acoustic": 0.72,
    "r&b_soul": 0.68,
    "rock": 0.45
}
```

用户无论上传什么音频，推荐结果永远相同。这意味着「基于声音特征的智能推荐」功能实际上并不存在。

**修复方向：** 使用 librosa 提取真实音频特征（音高范围、音色亮度、语速等），训练或规则匹配风格分类器。

---

### 5. ACE-Step 与 Fish Speech 存在双人声叠加

ACE-Step 是一个端到端的歌曲生成模型，其输出的完整歌曲**已包含人声**。项目又额外用 Fish Speech 生成了一轨说话人声，混音时会出现两套人声叠加的问题。

**正确做法：**

- 方案 A：让 ACE-Step 以 `instrumental_only` 模式只生成纯器乐伴奏，再叠加合成人声
- 方案 B：保留 ACE-Step 完整输出，用 SVC 对其中的人声轨做音色转换

---

### 6. 声音克隆逻辑存在根本性错误

```python
# 错误用法
embedding = vqgan.encode(audio)  # VQGAN编码
```

Fish Speech 的 VQGAN 编码输出是**离散 token 序列**，用于声码器解码，不是连续的声音风格嵌入向量。将其保存后再加载传回合成接口，在语义上是错误的。

**Fish Speech 正确的零样本克隆流程：** 将参考音频以 `reference_audio` 的形式直接传入推理接口，模型内部完成风格迁移，无需手动提取并保存中间嵌入。

## 🟡 工程问题

### 7. 采样率不统一导致音质损失

| 模块 | 采样率 |
|------|--------|
| ACE-Step 输出 | 48,000 Hz |
| Fish Speech 输出 | 24,000 Hz |
| 混音处理统一到 | 24,000 Hz |

混音时将 ACE-Step 的 48kHz 伴奏重采样到 24kHz，会导致高频细节丢失、音色劣化。应统一以 44.1kHz 或 48kHz 为工作采样率进行全链路处理。

---

### 8. Celery 任务队列配置了但未生效

`docker-compose.yml` 中配置了 Redis + Celery，但 `/api/v1/song/generate` 接口是**同步执行**的——生成任务直接在请求线程中运行，不走任务队列。

这导致：
- 生成期间（2~4 分钟）前端请求长时间阻塞
- 无法支持多用户并发
- 服务器崩溃时任务结果丢失

---

### 9. GPU 显存峰值估算偏乐观

| 组件 | 显存占用 |
|------|---------|
| ACE-Step | ~12 GB |
| Fish Speech | ~6 GB |
| 同时加载峰值 | **~18 GB** |

文档标注"12-16GB"，实际峰值超出消费级 GPU（16GB）上限，在 RTX 4080 等显卡上会直接 OOM。

## ✅ 推荐改造方案

### 方案 A（推荐）：ACE-Step 端到端演唱 + SVC 音色转换

```
用户音频
    ↓
RVC / So-VITS-SVC 训练用户音色模型
    ↓
ACE-Step 生成完整演唱歌曲（含人声）
    ↓
提取人声轨（Demucs 分离）
    ↓
SVC 将人声转换为用户音色
    ↓
与原伴奏轨重新混音输出
```

**优点：** 人声本身就是演唱腔，旋律、节拍、咬字天然对齐，只需替换音色。

---

### 方案 B：SVS 模型替代 Fish Speech

```
用户音频
    ↓
提取音色特征
    ↓
ACE-Step 生成纯伴奏（instrumental 模式）+ 旋律 MIDI
    ↓
DiffSinger / VITS-SVS 根据 MIDI + 歌词 + 用户音色生成演唱人声
    ↓
对齐混音输出
```

**优点：** 人声与旋律精确对齐，可精细控制每个音符。  
**缺点：** 流程更复杂，SVS 模型对歌词时间戳标注要求较高。

---

## 总结

项目的整体工程架构是扎实的，FastAPI + Pipeline 编排 + 模块化设计这套骨架完全值得复用。当前无法生成真实演唱歌曲，根本原因是**把 TTS 模型当 SVS 模型用**，这是一个技术选型层面的问题，需要在人声合成这一环做根本性替换，而不需要推倒重来。

优先修复顺序建议：

1. **替换人声合成模型**（致命，影响核心功能）
2. **实现人声-旋律对齐机制**（致命，影响音乐性）
3. **修复声音克隆逻辑**（严重，影响克隆效果）
4. **启用 Celery 异步任务**（严重，影响并发能力）
5. **统一采样率**（一般，影响音质）
6. **实现真实风格推荐算法**（一般，影响用户体验）
